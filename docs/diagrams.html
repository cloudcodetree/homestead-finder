Key pieces:
  • scraper/main.py is the entry point the cron job calls. It runs each registered source through the fetch-strategy chain (see next diagram), scores each listing, and writes two JSON files.
  • data/listings.json (root) is the live dataset the frontend reads. listings_YYYYMMDD.json is an append-only archive.
  • Scores ≥ 75 get batched into a SendGrid email via notifier.py.
  • Any commit to main (including the daily scrape commit) fires deploy-pages.yml, which builds the Vite bundle and pushes to Pages.
  • The React app doesn't have a backend — it just fetch()s data/listings.json from the same origin at page load.

Why this design (ADR-008, ADR-009):
  • Tiers 1 and 2 are free. LandWatch currently blocks both (Cloudflare detects headless Playwright), so real runs fall through to Tier 3.
  • Firecrawl is the workhorse for paid runs. FIRECRAWL_API_KEY is in GitHub secrets; parser runs on its markdown output.
  • Claude extraction (Tier 4) is last-resort — only triggered when even Firecrawl returns unparseable content. The Model Escalator starts at Haiku and bumps up on failure, capped at $1/run.
  • Learned selectors are git-committed so Tier 0 stays cheap over time: pay once for discovery, reuse forever until selectors break.
  • Today, only the LandWatch source has its parser wired up to the markdown output; lands_of_america still needs to be ported (P1 on the backlog).

What this buys (and costs):
  • Zero ongoing API spend. All three AI features bill against the Max flat fee via claude -p. CI stays on parsing only.
  • Idempotent enrichment. enrich.py hashes each listing; unchanged listings skip re-enrichment. Re-runs are effectively free thanks to data/cache/llm/.
  • Curation is two-stage — a deterministic pre-rank (0.4·dealScore + 0.6·homesteadFit − 5·redFlags) picks the top 50, then one Sonnet call chooses and explains the top N.
  • Vocabularies are duplicated between scraper/ai_vocab.py and frontend/src/types/property.ts — kept in sync by hand (see ADR-012). emit_ts_vocab.py is the codegen path if this becomes annoying.
  • Ask-Claude only works in npm run dev. The React app pings /health; if the localhost proxy isn't running (as on production Pages), the bar silently hides.
  • Tradeoff: AI data only refreshes when the developer manually runs enrich / curate and commits. No automation — that's the price of free.

4. Deal Scoring Formula

Every listing gets a dealScore from 0 to 100. Scores ≥ 75 trigger email alerts. Here's the math, straight out of scoring.py (ADR-003).

pie showData title Weight of each factor (max points) "Price vs. regional median" : 40 "Homesteading features" : 30 "Days on market" : 20 "Source reliability" : 10
Reading the numbers:
  • Price (40%) — the dominant signal. A $400/acre listing in Montana (median $450) scores near the top; the same price in California (median $5,000) would too, but won't exist.
  • Features (30%) — water_well alone is worth 8 points, reflecting how non-negotiable water is for a working homestead. Feature detection depends on listing text quality.
  • DOM (20%) — longer on market = more seller flexibility. The curve rewards ~180+ day listings.
  • Source (10%) — county_tax and govease top out because tax-sale and government auction lots are where the actual deals hide. MLS feeds (Zillow, Realtor) cap at 4–5 since they're retail-priced.
  • ADR-003 says to revisit these weights after 30 days of real data. If too many listings cluster in 40–60, the curve gets adjusted. That review hasn't happened yet because real data only started flowing with the LandWatch Firecrawl fix on 2026-04-20.
